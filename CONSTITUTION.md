# CONSTITUTION DE MOMENTUM

> *« Une application devrait donner envie d'être utilisée.
> Momentum doit donner envie d'aller vivre dehors. »*

Version 1.0

---

# Lettre du fondateur

Bienvenue.

Si tu lis ce document, c'est que tu participes à la construction de Momentum.

Peu importe que tu sois un développeur, une intelligence artificielle ou un futur collaborateur. Tu fais désormais partie de l'équipe.

Momentum est né d'un constat simple.

Le monde du sport produit toujours plus de données, de graphiques et de statistiques. Pourtant, beaucoup de sportifs finissent par regarder davantage leurs chiffres que leurs souvenirs.

Je ne veux pas construire une application qui enferme les utilisateurs dans un écran.

Je veux construire une application qui leur donne envie de le refermer.

Chaque ligne de code, chaque écran et chaque fonctionnalité doivent servir cette idée.

Je préfère supprimer une fonctionnalité plutôt que d'alourdir l'expérience.

Je préfère attendre une semaine de plus plutôt que publier quelque chose dont nous ne serons plus fiers dans un an.

Nous ne cherchons pas à devenir l'application qui fait le plus de choses.

Nous voulons devenir celle qui fait les bonnes choses.

Merci de prendre soin de ce projet.

— Christophe Gyger

# Notre mission

Transformer des données en motivation.

Transformer des statistiques en histoires.

Transformer des entraînements en aventures.

Les données existent pour servir l'humain.

Jamais l'inverse.

---

# Notre philosophie

> **L'esthétique d'un magazine d'aventure.
> La rigueur d'un outil d'entraînement.**

Cette phrase doit guider toutes les décisions du projet.

Si une idée ne respecte pas ces deux dimensions, elle doit être repensée.

---

# Ce qu'est Momentum

Momentum accompagne une personne dans son chemin.

Pas seulement dans sa pratique sportive.

Le sport, le bien-être et les aventures du quotidien racontent ensemble une histoire.

L'application doit conserver cette vision globale.

---

# Ce que Momentum n'est pas

Momentum n'est pas :

- un réseau social classique ;
- un tableau Excel sportif ;
- un clone de Strava ;
- un clone de Garmin Connect ;
- une application médicale.

Nous ne cherchons pas à afficher toutes les données disponibles.

Nous cherchons à afficher les données qui ont du sens.

---

# Notre manière de concevoir

Avant d'ajouter une fonctionnalité, toujours se poser trois questions.

Est-elle utile ?

Est-elle compréhensible ?

Améliore-t-elle réellement l'expérience ?

Si une seule réponse est négative, la fonctionnalité doit être retravaillée.

La simplicité est une qualité.

Supprimer est parfois une meilleure décision qu'ajouter.

---

# Les données

Les données ne sont jamais une finalité.

Une donnée est affichée uniquement parce qu'elle aide l'utilisateur à mieux comprendre son parcours ou à prendre une meilleure décision.

Une métrique impressionnante mais inutile n'a pas sa place dans Momentum.

---

# Le design

Le design doit respirer.

Il inspire le calme, la confiance et l'envie de sortir.

Chaque écran doit pouvoir être regardé longtemps sans fatigue.

Nous recherchons une élégance discrète.

Pas un effet spectaculaire.

Nos inspirations sont davantage les magazines d'aventure, les beaux livres et les objets bien dessinés que les tableaux de bord techniques.

---

# Les histoires avant les performances

Une sortie de dix kilomètres peut être plus importante qu'un ultra.

Un coucher de soleil peut avoir autant de valeur qu'un record personnel.

Momentum ne célèbre pas uniquement la performance.

Il célèbre la progression et les souvenirs.

---

# Les décisions

Une bonne décision est une décision qui sera encore pertinente dans plusieurs années.

Nous privilégions toujours le long terme.

Une solution temporaire est acceptable uniquement si elle est clairement identifiée comme telle.

La dette technique n'est jamais créée par confort.

---

# Le code

Le code est un moyen.

Jamais une fin.

Il doit être :

- lisible ;
- modulaire ;
- documenté lorsque nécessaire ;
- compréhensible plusieurs mois plus tard.

Une personne qui découvre le projet doit pouvoir comprendre rapidement son organisation.

---

# Le dépôt GitHub

Le dépôt est la mémoire de Momentum.

Les conversations permettent de réfléchir.

Le dépôt conserve les décisions.

Toute évolution importante doit laisser une trace dans la documentation.

Le code seul ne raconte jamais toute l'histoire.

---

# Les intelligences artificielles

Les IA ne sont pas de simples exécutants.

Elles sont encouragées à réfléchir.

Si une meilleure solution existe, elles doivent la proposer.

Si une demande complexifie inutilement le produit, elles doivent le signaler.

Leur responsabilité est de protéger la qualité du projet autant que de produire du code.

---

# Les principes d'ingénierie

Les règles suivantes s'appliquent à toute contribution.

## Priorités

1. Préserver les données existantes.
2. Préserver le comportement actuel.
3. Préserver la compatibilité du projet.
4. Réaliser des modifications simples, explicites et facilement réversibles.

## Architecture

- Éviter les modifications structurelles inutiles.
- Conserver une architecture claire.
- Mettre à jour immédiatement les imports, liens et références après un renommage.
- Préférer plusieurs petits changements à une transformation massive.

## Données

- Ne jamais exposer de secret ou de clé privée.
- Ne jamais modifier une migration déjà appliquée.
- Créer une nouvelle migration pour toute évolution de la base.
- Respecter les politiques de sécurité et de confidentialité.

## Qualité

Avant toute livraison :

- vérifier les impacts ;
- préserver le travail existant ;
- tester les parcours concernés ;
- documenter les décisions importantes ;
- documenter les évolutions visibles ;
- signaler ce qui n'a pas pu être vérifié.

# Notre ambition

Nous voulons construire un produit qui vieillira bien.

Pas une démonstration technique.

Pas une accumulation de fonctionnalités.

Un compagnon de route.

Si, dans quelques années, un utilisateur ouvre Momentum avec le sourire avant d'aller courir, marcher, grimper ou simplement partir découvrir un nouvel endroit...

...alors nous aurons réussi.
